Muabilité

1. L’énigme

1.1. Point de départ

Le but est de comprendre le phénomène constaté :

from random import randrange

class Carte():
  valeurs = list(range(7,11)) + ['Valet','Dame','Roi','As']
  couleurs = ['Coeur','Carreau','Pique','Trèfle']

  def __init__(self,v=None,c=None):
    if not v:
      v = Carte.valeurs[randrange(0,len(Carte.valeurs))]
    if not c:
      c = Carte.couleurs[randrange(0,len(Carte.couleurs))]

    if not v in Carte.valeurs:
      raise ValueError(f"{v}: valeur incorrecte")
    if not c in Carte.couleurs:
      raise ValueError(f"{c}: couleur incorrecte")
    self.couleur = c
    self.valeur = v

  def __str__(self):
    return f"{self.valeur} de {self.couleur}"

  def __repr__(self):
    return f"{self.valeur} de {self.couleur}"

class Cartes():
  def __init__(self,c=None):
    if c:
      self.cartes = c.cartes
    else:
      self.cartes = []

  def __str__(self):
    return ", ".join([str(c) for c in self.cartes])

  def __repr__(self):
    return ", ".join([str(c) for c in self.cartes])

  def pioche(self):
    try:
      k = randrange(0,len(self.cartes))
    except ValueError:
      raise ValueError(f"pioche vide !")

    carte = self.cartes[k]
    self.cartes.remove(carte)
    return carte

  def ajoute(self,c):
    self.cartes += [c]

if __name__=="__main__":
  des_cartes = Cartes()

  for i in range(4):
    des_cartes.ajoute(Carte())

  les_memes = Cartes(des_cartes)

  print(f"{des_cartes=}\n {les_memes=}\n")
  des_cartes.pioche()
  print(f"{des_cartes=}\n {les_memes=}")

donne :

les_cartes=Valet de Carreau, 9 de Pique, 7 de Coeur, 8 de Trèfle
 les_memes=Valet de Carreau, 9 de Pique, 7 de Coeur, 8 de Trèfle

des_cartes=9 de Pique, 7 de Coeur, 8 de Trèfle
 les_memes=9 de Pique, 7 de Coeur, 8 de Trèfle

Pourquoi la modification de des_cartes influe sur les_mêmes ?

1.2. Reformulation

Un exemple simple permet de mettre en évidence le phénomène :

a = [1,2]
b = a
print(a,b)    # [1, 2] [1, 2]
a += [3]
print(a,b)    # [1, 2, 3] [1, 2, 3]

La modification de a impacte là aussi b. Pourquoi ?

2. Explication

On trouve l’explication en traçant ce qu’il se passe :

a = [1,2]
b = a
print(a is b)       # True
print(id(a),id(b))  # 140245663171968 140245663171968

Donc a et b référencent la même donnée.

a += [3]            (1)
print(a is b)       # True
print(id(a),id(b))  # 140245663171968 140245663171968
1 cette instruction modifie la liste [1,2].

a et b référencent toujours la même donnée, qui vient d’être modifiée.

On remarque au passage qu’une liste peut donc changer de valeur, c’est pourquoi dans la terminologie pythonique, le type list est dit muable (mutable).

3. Généralisation ?

Tous les type de données se comportent-ils de cette façon ? Comment réagit par exemple le type string ?

a = "uv"
b = a
print(a,b)    # uv uv
a += "w"
print(a,b)    # uvw uv

On constate que le comportement est différent, la modification de a n’a ici pas d’impact sur b. Explication :

a = "uv"
b = a
print(a is b)       # True
print(id(a),id(b))  # 140245667117360 140245667117360

A ce stade, comme précédemment, a et b référencent la même donnée.

a += "w"            (1)
print(a is b)       # False
print(id(a),id(b))  # 140245667117392 140245667117360
1 cette instruction ne modifie pas la donnée uv, mais en crée une nouvelle uvw.

Après la modification de a, a et b ne référencent plus la même donnée.

L’explication tient au fait qu’en Python, une chaîne de caractères n’est pas modifiable. Dans la terminologie pythonique, le type str est dit immuable (immutable). Ce qui semble être une modification est en réalité une création.

Il existe 2 catégories de type en Python, les types muables et les immuables.

4. Récapitulatif

Les objets dont la valeur peut changer sont muables, ceux dont la valeur est définitivement fixée à leur création sont immuables. Par exemple :

t = (1,2,3)
print(t, id(t))  # (1,2,3) 139727011356608
t += (4,5)
print(t,id(t))   # (1,2,3,4,5) 139727011374448

On constate ci-dessus que la concaténation d’un tuple à t crée une nouvelle donnée, puisque l’id de t change. Donc le type tuple est immuable. Mais le tuple peut avoir un (ou plusieurs) élement lui-même muable, par exemple un élément de type list :

t = (1,2,[3,4])
print(t, id(t))      # (1,2,[3,4]) 139727011443776
print(t[2],id(t[2])  # [3,4] 139727011443712

et dans ce cas, l’élément muable peut changer de valeur :

l = t[2]
l += [5]
print(t,id(t))   # (1,2,[3,4,5]) 139727011443776
print(t[2],id(t[2])  # [3,4,5] 139727011443712

Donc au premier niveau, les éléments du tuple sont figés, aux autres niveaux, les données peuvent changer.

5. Quel impact ?

5.1. Références multiples

Méfiance lorsque 2 variables pointent vers la même donnée :

a = "uv"
b = a

Les strings étant immuables, a et b auront ici leur propre ligne de vie. Ici a is b, mais après une modification de a ou de b, a is b sera False. Par contre, les listes étant muables :

a = [1,2]
b = a

alors a et b partageront toujours la même ligne de vie : a is b, et le restera après toute modification de a ou b.

copy()

La fonction copy du module copy permet de défaire cette liaison :

from copy import copy

a = [1,2]
print(a,b,a is b)       # [1, 2] [1, 2] True
b = copy(a)
print(a,b,a is b)       # [1, 2] [1, 2] False
b += [3]
print(a,b)              # [1, 2] [1, 2, 3]

5.2. Arguments de fonction

Attention aux arguments de fonction, au retour de l’appel :

def f(x,y):
  x += y
  print(f"{x=}") ;

a = "uv"
f(a,"w")             # x='uvw'
print(f"{a=}")       # a='uv'

b = [1,2]
f(b,[3])             # x=[1, 2, 3]
print(f"{b=}")       # b=[1, 2, 3]

On constate ici que l’argument a de type str est intact après exécution de f(), alors que l’argument b de type list a été modifié. Pour être sûr qu’une fonction ne va pas modifier un argument muable, mieux vaut envoyer une copie :

def f(x,y):
  x += y

a = [1,2]
f(copy(a),[3])
print(a)       # [1,2]
